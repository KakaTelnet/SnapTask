# snap-goal-make 设计

## 状态

本设计已经过用户逐节确认。下一阶段仅根据本文编写实施计划；本文不代表 Skill 已实现。

## 目标

新增通用 AI Agent Goal 生成 Skill `snap-goal-make`。它通过多轮对话把模糊意图转化为结构完整的 Goal，并在软件项目中优先读取授权的项目规则、代码、接口、测试和文档。

`snap-goal-make` 负责生成与修订，`snap-goal-review` 是唯一质量裁决者，`task-decompose` 只接收审核通过的 Goal。

## 核心边界

- Maker 不评分，不执行 Reviewer 的八项硬门槛，不输出 `Approved`、`Revision Required` 或 `Rejected`。
- Maker 可以维护轻量的信息收集状态，用于决定下一步问什么，但不能把它变成内部审核器。
- 完整 Goal 必须经过用户确认后才能交给 `$snap-goal-review`。
- Reviewer 返回 `Approved` 后，Maker 不得再次改写 Goal。
- 不使用补丁式修订，不保留 Goal 版本号。
- 不增加 Reviewer 返回字段；直接使用现有的“问题、影响、修改要求”等输出。

## 组件

```text
skills/snap-goal-make/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    └── goal-template.md
```

- `SKILL.md`：定义适用场景、多轮对话协议、上下文读取规则、草稿确认和跨 Skill 交接。
- `references/goal-template.md`：定义固定骨架，以及结果型 Goal 和有边界调查型 Goal 的表达方式。
- `agents/openai.yaml`：定义显示名称、简短描述和默认提示。
- `skills/snap-goal-review/SKILL.md`：将修订交接对象从 `$snap-goal` 调整为 `$snap-goal-make`。
- `.codex-plugin/plugin.json`：将 Goal 创建能力加入插件描述和默认提示。

## 对话架构

### 1. Intake

接收模糊想法、现有 Goal 草稿或 Reviewer 报告，判断当前入口。识别用户授权的资料范围，不把未授权信息作为事实来源。

### 2. Context

通用场景读取用户提供或授权的资料。软件项目优先读取 `AGENTS.md`、相关代码、接口、测试和项目文档。能够从资料中确认的信息不重复询问用户。

### 3. Discovery

先判断是结果型 Goal 还是有边界调查型 Goal。每轮解决一个会影响结果、验收、边界、权限、风险或取舍的主要决策。

每轮使用简短的渐进摘要：

```text
已确认：
暂定假设：
待确认：
```

关键偏好必须询问。低风险细节可以合理推断，但必须标记为假设。资料冲突时必须展示冲突并请求确定优先级或升级规则。

### 4. Draft

信息足够后，按照固定骨架生成一份完整 Goal。用户确认前不得送审。用户明确跳过审核时，只能输出 `Unreviewed Draft`，不能宣称已经适合执行，也不由 Maker 主动交给任务拆解。用户坚持绕过审核时，由 `task-decompose` 按其现有的显式 bypass 规则处理。

### 5. Review

把用户确认的完整 Goal 原样交给 `$snap-goal-review`。Maker 的推理过程、推荐倾向和对通过的期待都不是审核依据。

### 6. Result Handling

```text
通过: Approved
→ 保持 Goal 原文不变
→ 连同审核结果和证据要求交给 $task-decompose

修订: Revision Required
→ 读取问题、影响和修改要求
→ 只追问受影响的关键决策
→ 重新生成一份完整 Goal
→ 用户确认后重新提交完整审核

驳回: Rejected
→ 保留仍有效的背景事实
→ 回到目标状态或调查边界的重新定义
→ 不做表面润色
```

相同问题连续两次没有获得新信息时暂停，请求用户作出决策，避免无效循环。

## Goal 固定骨架

所有 Goal 使用固定骨架并按复杂度控制篇幅。简单 Goal 可以每部分一句；复杂 Goal 必须保留足够细节。不适用项需说明原因，不能静默省略。

```text
Goal 类型：
结果型 | 有边界调查型

背景与价值：
当前状态、受影响对象、完成价值。

目标：
使目标对象从当前状态达到可观察的最终状态。
或在明确范围和限制内完成调查并交付可验证产物。

验收标准：
- AC-1：条件、对象、预期结果
- AC-2：条件、对象、预期结果

完成证据：
| 验收编号 | 所需证据 | 验证方式 |

范围与权限：
- 包含
- 不包含
- 允许读取或修改
- 禁止事项

约束与优先级：
- 必须保持的规则、数据、兼容性或安全属性
- 时间、成本或调用次数限制
- 冲突时的取舍顺序

事实来源：
- 权威代码、文档、接口或数据
- 来源冲突时的优先级和升级方式

假设与未知项：
- 已采用的假设
- 尚未确认的信息

停止与升级条件：
- 完成条件
- 无进展阈值
- 高风险、越权、不可逆或来源冲突时的升级规则

完成报告要求：
- 已完成
- Unfinished
- Unverifiable
- Blocked
- Out of scope
- Residual risks
```

## 特殊 Goal 类型

结果型 Goal 必须描述可观察的最终状态，不能只写动作或“尽力改进”。

有边界调查型 Goal 不必保证找到根因或答案，但必须明确：

- 调查对象和范围
- 必须检查的资料或路径
- 可验证交付物
- 时间、成本或尝试次数限制
- 完成和停止条件

角色视角不是必填项。只有角色能够约束权限边界、取舍标准或 Review 视角时才加入 Goal；角色名称本身不构成有效约束。

## 异常处理

- 资料无法访问：明确说明缺失内容，不假装已经读取。
- 资料相互冲突：列出来源和冲突点，请用户确定优先级或升级规则。
- 关键需求不明确：继续追问，不把建议写成用户已确认事实。
- 目标范围过大：提出拆分建议，由用户确认 Goal 边界后再继续。
- Reviewer 不可用：输出完整 Goal 和可复用的审核交接提示，不伪造审核结果。
- 用户要求强行通过：仍然交给 Reviewer 独立判断，Maker 不降低验收或证据要求。

## 验证设计

新增 `tests/goal-make-cases.json`，至少覆盖：

1. 模糊结果型目标先追问再生成。
2. 软件项目主动读取授权上下文。
3. 有边界调查目标不承诺未知发现。
4. 低风险假设被明确标注。
5. 事实来源冲突不被静默解决。
6. 过大目标先请求确认拆分边界。
7. 现有 Goal 草稿只补关键缺口。
8. `Revision Required` 触发定向追问和完整重生成。
9. `Rejected` 触发目标重定义。
10. `Approved` Goal 原样交给 `$task-decompose`。
11. 跳过审核时只输出 `Unreviewed Draft`。
12. 角色视角仅在确有边界或评价价值时出现。
13. Maker 不评分、不输出 Reviewer 的三态裁决。
14. 固定骨架能够根据复杂度保持合适篇幅。

验证层次包括：

- Skill front matter、相对引用和插件结构校验
- 固定输出契约检查
- 多轮对话行为测试
- Maker、Reviewer 和 Task Decompose 的交接测试
- 隔离安装后的三个 Skill 发现和触发测试

## 非目标

- 不实现独立运行的工作流服务或持久化状态机。
- 不建立第二套 Goal 评分或审核规则。
- 不自动创建任务卡；任务拆解仍由 `$task-decompose` 负责。
- 不引入 Goal 版本号、补丁协议或程序化 JSON 返回格式。
- 不在本次工作中重构 Reviewer 的评分体系。

## 完成标准

- 用户可以从模糊意图开始，通过多轮对话得到完整 Goal。
- 软件项目中，Maker 能利用授权的本地上下文减少无效提问。
- Goal 使用固定骨架并支持结果型和有边界调查型两类表达。
- Maker 不承担质量裁决，完整 Goal 能可靠交给 `$snap-goal-review`。
- Reviewer 的三种结果能够分别驱动任务拆解、完整修订或目标重定义。
- 三个 Skill 在插件中可发现，交接名称一致，行为测试通过。
