# snap-goal-review 中文说明

> 本文件是 `snap-goal-review` 的中文讲解版，便于人阅读和维护。
> Codex 自动发现与执行时仍以 `SKILL.md` 和 `references/review-rubric.md` 为准。

## 这个 Skill 是什么

`snap-goal-review` 是 SnapTask 中负责审核 Goal 质量的 Skill。它站在独立 Reviewer 的位置，判断一个 Goal 是否已经清楚到可以进入规划、任务拆解、执行或交接。

它不负责默认生成 Goal，也不默认重写 Goal。它的工作是判断：

- Agent 是否知道“完成”是什么意思；
- Agent 是否能拿出证据证明已经完成；
- Agent 是否知道可以改什么、不能改什么；
- Agent 是否知道什么时候停止、升级或请求人工确认；
- 如果只完成了 80%，Agent 是否能识别剩余 20%。

核心原则：

```text
不要写“尽力做某事”，而要写“达到什么状态才算完成”。
```

## 什么时候使用

在以下情况使用 `snap-goal-review`：

- 一个 AI Agent Goal 需要进入执行前审核；
- 一个项目目标、成功定义或验收标准需要检查；
- 一个调研型 Goal 需要判断边界、交付物和停止条件；
- 一个 Goal 准备交给 `task-decompose` 拆任务卡；
- 你想确认 Goal 是否可执行、可验收、可交接。

不适合用它做的事：

- 直接替用户创作完整 Goal；
- 默认重写 Goal；
- 替 Goal 作者补充未声明的需求；
- 为了让流程继续而降低审核标准。

## 角色边界

`snap-goal-review` 的默认行为是 **只审核，不重写**。

它必须避免：

- 发明需求；
- 把假设当成事实；
- 因为用户希望通过就提高结论；
- 在交接时悄悄降低验收、边界、约束、证据或停止规则。

如果用户明确要求提供参考修订版，Reviewer 也必须先完成独立审核，再把参考修订版放在单独的最后部分。

## 审核流程

1. 读取 Goal，以及用户授权的上下文、事实来源和证据。
2. 区分已声明事实、假设、未知项和缺失证据。
3. 读取 `references/review-rubric.md`，先应用所有硬门槛，再评分。
4. 只能选择一个审核结果：
   - `通过: Approved`
   - `修订: Revision Required`
   - `驳回: Rejected`
5. 将每条 Acceptance 映射到它需要的证据。
6. 对每个问题写清楚：问题、影响、修改要求。
7. 以一个明确的下一步动作结束。

## 三种审核结果

### `通过: Approved`

表示 Goal 已经可以进入规划或任务拆解。

必须同时满足：

- 8 个硬门槛全部满足；
- 综合评分为 10-12 分；
- 没有未解决的矛盾；
- 没有危险的权限或安全缺口；
- 验收标准和证据足以判断完成。

交接方式：

```text
将未修改的 Goal、审核结果和证据要求交给 task-decompose。
```

### `修订: Revision Required`

表示 Goal 的方向可用，但还有可修正的缺口，暂时不能通过。

常见原因：

- 结果状态还不够可观察；
- 验收标准存在，但不够客观；
- 有测试或证据要求，但没有一一映射；
- 范围、权限、事实来源或停止条件不完整；
- 剩余工作报告要求不够清楚。

交接方式：

```text
返回 snap-goal-make 或 Goal 作者修订，不生成最终任务卡。
```

### `驳回: Rejected`

表示 Goal 没有可用的完成基础，需要重新定义。

常见原因：

- 只有“尽力做”“优化一下”“改进体验”这类努力型描述；
- 没有可观察结果；
- 没有可用验收标准；
- 目标内部矛盾、不可执行或不安全；
- Reviewer 必须发明 Goal 才能继续审核。

交接方式：

```text
停止拆解，请求重新定义 Goal。
```

## 8 个硬门槛

只有 8 个硬门槛全部满足时，才可能得到 `通过: Approved`。任何一项为 `需要修订` 或 `暂无法判断`，都不能通过。

| 序号 | 门槛 | 检查重点 |
| --- | --- | --- |
| 1 | 完成形态 | Outcome Goal 必须有目标对象和可观察最终状态；调研型 Goal 必须有调查边界、交付物、证据、限制和停止点。 |
| 2 | 可检查验收 | 必须有客观条件判断是否完成。 |
| 3 | 证据映射 | 每条验收标准都必须有对应证明，不能只写“充分测试”。 |
| 4 | 边界与权限 | 必须区分包含内容、排除内容、允许修改内容和禁止事项。 |
| 5 | 约束与优先级 | 必须保留关键不变量；当时间、成本、质量、范围或安全冲突时，要说明优先级。 |
| 6 | 事实来源优先级 | 必须说明以哪些代码、文档、接口或数据为准，以及来源冲突时怎么处理。 |
| 7 | 停止与升级 | 必须定义完成条件、风险操作确认、无进展阈值和升级规则。 |
| 8 | 剩余工作可见性 | 完成报告必须能区分 Unfinished、Unverifiable、Blocked、Out of scope、Residual risks。 |

硬门槛结果只能使用：

- `已满足`：明确且可用；
- `需要修订`：存在但模糊、不完整或连接弱；
- `暂无法判断`：缺少上下文，无法公平判断。

## 横向检查

审核时还需要额外检查这些问题：

- **努力型语言**：如 try、strive、improve、appropriate、best effort，必须转化为可衡量终态。
- **过度规定实现**：实现细节只能用于保护边界或不变量，不能代替结果判断。
- **内部一致性**：验收、范围、约束、事实来源和停止规则不能互相矛盾。
- **权威压力**：用户要求通过，不能覆盖审核规则。
- **角色有效性**：角色只有在明确边界、评价标准或 review 视角时才有价值；只有头衔没有分数。

## 评分规则

总分 12 分，共 6 个维度，每项 0-2 分。

| 分数 | 含义 |
| --- | --- |
| 0 | 缺失、矛盾或不可用 |
| 1 | 存在但模糊、部分满足或连接弱 |
| 2 | 清楚、一致、可观察、可执行 |

评分维度：

1. **结果清晰度**：目标对象、最终状态或有边界调研是否清楚；必要时是否有背景价值。
2. **可验收性**：是否有客观验收条件和完成判断。
3. **证据质量**：证据是否具体，并与验收一一对应。
4. **边界清晰度**：范围、非目标、权限、事实来源和冲突处理是否明确。
5. **风险控制**：不变量、优先级、限制、无进展规则和升级条件是否完整。
6. **完整性可见性**：是否能报告假设、未知项和五类剩余工作。

结果选择规则：

- `通过: Approved`：10-12 分，且 8 个硬门槛全部满足。
- `修订: Revision Required`：方向可用，但有一个或多个可修正缺口。
- `驳回: Rejected`：没有可观察结果、没有可用验收基础、目标不可能或不安全，或必须由 Reviewer 发明 Goal 才能继续。

注意：评分不能覆盖硬门槛。高分但硬门槛失败时，只能是 `修订: Revision Required`，不能是 `通过: Approved`。

## 输出格式

审核输出应使用用户语言，但三种结果标签必须逐字保留：

```text
审核结果: 通过: Approved | 修订: Revision Required | 驳回: Rejected
综合评分: N/12

总体判断:
<一段简短判断>

做得好的地方:
- <基于证据的优点，或“无”>

关键检查:
| 检查项 | 结果 | 依据 |
| ... | 已满足 / 需要修订 / 暂无法判断 | ... |

Acceptance-to-evidence map:
| Acceptance | Required evidence | Mapping |
| ... | ... | Direct / Partial / Missing |

需要改进的地方:
1. 问题:
   影响:
   修改要求:

假设与未知项:
- <条目，或“无”>

Remaining work and risks:
- Unfinished:
- Unverifiable:
- Blocked:
- Out of scope:
- Residual risks:

下一步:
- <一个具体动作>
```

对于 Goal 草稿，`Remaining work and risks` 不是在判断执行是否已经开始，而是在判断这个 Goal 是否要求执行结束时暴露这些类别。

## Finding 写法

每个问题必须包含三部分：

```text
问题: 缺失、模糊、矛盾或没有证据支持的内容
影响: 它如何影响执行、验收、安全或完成判断
修改要求: Goal 作者必须补充或澄清的信息
```

Reviewer 可以指出缺口，但不能默认替作者做决定。

## 与 SnapTask 其他 Skill 的关系

推荐流程：

```text
Goal 草稿
  -> snap-goal-review 审核
  -> 通过: Approved
  -> task-decompose 拆任务卡
  -> Agent 执行
  -> 证据回收
  -> Goal 完成判断
```

当审核结果不是 `通过: Approved` 时：

- `修订: Revision Required`：回到 Goal 作者或 `snap-goal-make` 修订；
- `驳回: Rejected`：停止拆解，重新定义 Goal。

## 调研型 Goal 的特殊处理

调研、排障、探索类 Goal 不一定能保证找到根因或最终答案，所以不能要求它承诺不可保证的发现。

但它仍然必须定义可验证的结束状态，例如：

- 调查范围；
- 输入资料；
- 时间或尝试次数限制；
- 需要输出的事实、排除项、可能原因和建议；
- 停止条件；
- 需要升级的情况。

可接受的调研型 Goal 不是：

```text
尽力调查登录失败问题。
```

而应类似：

```text
在 4 小时内检查指定日志、认证调用链和最近变更，
输出已验证事实、被排除假设、最可能原因、证据和下一步建议；
如果缺少生产日志权限或连续 2 次尝试没有新证据，则暂停并请求人工决策。
```

## 常见错误

- 默认重写 Goal，而不是返回修改要求；
- 把语言流畅或表达自信当成证据；
- 用综合评分覆盖失败的硬门槛；
- 因为调研 Goal 不能保证找到根因就直接驳回；
- 要求过多实现细节，而不是要求可观察结果；
- 发现上下文缺失时，只说“不清楚”，没有说明影响和解决方式；
- 在交给 `task-decompose` 时降低验收标准或省略风险。

## 最小使用示例

```text
$snap-goal-review

请只审核，不要重写。

Goal:
让用户能够通过邮箱完成登录，并在凭据无效、网络异常和账户禁用时获得明确反馈。

验收标准:
1. 正确凭据可进入既有登录后页面，并由现有集成测试验证。
2. 空字段显示必填提示且不提交，并由 UI 测试验证。
3. 无效凭据显示安全的通用错误，并由集成测试验证。
4. 网络失败显示重试操作，并由 UI 测试验证。

范围:
包含 web/auth 和对应测试。
不包含后端 API、注册、密码重置和视觉重做。

约束:
不得修改认证协议；不得降低账号枚举保护；只能修改 web/auth。

事实来源:
以现有认证 API schema 和登录集成测试为准；冲突时先停止并报告。

停止与升级:
四条验收全部有通过证据时完成；连续 2 次尝试没有新证据、没有减少阻塞且没有合理替代路径时暂停；涉及后端、安全策略或破坏性修改时请求确认。
```
