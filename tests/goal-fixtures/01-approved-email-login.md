# 审核输入内容

## Goal

让现有邮箱登录表单对本 Goal 明确定义的四种提交状态都产生可观察结果，同时保留当前认证语义。这四种状态构成本 Goal 的完整集合：有效凭据、客户端空字段、无效凭据、客户端网络传输失败。

## 背景

用户在登录过程中收到的反馈不一致。本次工作应提升反馈清晰度，但不得改变认证契约、跳转逻辑或账号枚举保护。

限流、服务端 5xx、账户禁用、会话冲突或 schema 中发现的其他状态不属于本 Goal 的“提交状态”集合；发现时记录为 Out of scope 和 Residual risks，不得静默扩大实现范围。

## 验收标准

1. 当用户输入有效凭据时，应进入现有登录后路由；通过当前登录集成测试验证。
2. 当邮箱或密码为空时，表单应显示必填提示且不提交；通过 UI 测试验证。
3. 当凭据无效时，表单应显示现有的安全通用错误；通过集成测试验证。
4. 当现有登录测试所模拟的客户端网络传输失败发生时，表单应显示重试操作；通过 UI 测试验证。其他未被当前测试契约定义的传输失败变体不在本 Goal 范围内。
5. 现有跳转和账号枚举回归测试保持通过。

## 范围

包含：

- `web/auth/`
- 现有登录测试，以及直接覆盖四种提交状态的新测试

不包含：

- 后端认证 API
- 注册
- 密码重置
- 登录表单之外的视觉重做
- 限流、服务端 5xx、账户禁用、会话冲突及其他未列出的认证状态

## 约束

- 只能修改 `web/auth/` 和相关测试。
- 必须保留当前认证契约、跳转逻辑和账号枚举保护。
- 质量和安全优先于速度。

## 事实来源

- 第一优先级：现有认证 API schema。
- 第二优先级：当前登录集成测试。
- 如果 schema 与测试冲突，停止并报告冲突，不要改变后端行为。

## 完成证据

- 每条验收标准对应的精确测试命令和通过输出。
- 变更文件列表，证明只修改了范围内文件。
- 每条验收标准与证据的简短映射说明。

## 停止与升级

- 只有所有验收标准都通过且证据完成映射时才算完成。
- 连续 2 次尝试没有新证据、没有减少阻塞且没有合理替代路径时停止。
- 涉及后端、schema、安全策略或破坏性修改前必须请求确认。

## 完成报告要求

报告以下类别：

- Unfinished
- Unverifiable
- Blocked
- Out of scope
- Residual risks
