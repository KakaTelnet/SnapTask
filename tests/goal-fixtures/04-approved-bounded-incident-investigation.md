# 审核输入内容

## Goal

在 4 小时内调查 `2026-07-14 10:00-11:00 UTC` 期间 `POST /login` 5xx 错误激增的问题，并产出有边界的事故报告。不保证一定找到根因。

## 调查边界

只检查：

- 事故时间窗口内的 `auth-api/error.log`
- 事故时间窗口内的 `gateway/access.log`
- 从 `POST /login` 到 `AuthService` 再到 token issuer 的调用链
- 从 `v2.3` 的 peeled commit（不含）到调查开始时记录的 `main` HEAD（含）之间，触及 `web/auth` 或 `backend/auth` 的最新 20 个非 merge commit。只选择 `main` 可达且 peeled commit 不可达的提交，按 Git committer timestamp 降序、完整 40 字符 commit hash 升序打破同时间戳并列。若 `v2.3` 是 annotated tag，报告同时记录完整 tag object hash 和 peeled commit hash，并仅用 peeled commit 计算区间；若是 lightweight tag，报告记录 tag ref 直接指向的完整 commit hash、注明不存在独立 tag object，并将该 commit hash 作为 peeled commit 计算区间。报告还记录 `main` HEAD 完整哈希和最终提交列表。

## 交付物

1. 已验证事实，包含来源、时间戳和证据。
2. 已排除假设，包含排除理由和支持证据。
3. 最可能原因，包含置信度和证据。
4. 未知项和缺失访问权限。
5. 建议的下一步。
6. 完成报告，区分 Unfinished、Unverifiable、Blocked、Out of scope 和 Residual risks。

## 验收标准

1. 每个指定日志来源、`POST /login` → `AuthService` → token issuer 的每一段调用链，以及按既定规则选出的提交范围都被检查；无法检查的项目以 Unverifiable 标记并说明原因。
2. 每个发现都映射到来源路径、时间戳或提交哈希，以及证据摘录或命令输出。
3. 报告中的每项判断都明确标记为已验证事实、假设或未知项，且事实与假设分栏呈现。
4. 调查开始前建立覆盖全过程的只读技术控制，并证明代码、日志、追踪、配置和生产数据未被本次调查修改；仅有调查前后快照不构成充分证明。

## 范围

包含：

- 对指定日志、调用链和提交的只读调查。
- 通过当前任务的最终响应渠道输出事故报告；这是唯一获准的写入渠道，不属于被调查系统。

不包含：

- 代码修改
- 配置修改
- 数据修复
- 生产缓解操作
- 将报告或中间材料写入仓库、日志系统、追踪系统、配置系统或生产数据系统

## 约束

- 只允许只读访问。
- 不得在报告中暴露密钥或个人数据。
- 指定日志和调用链数据只能通过现有 incident-reader 只读角色以最小必要、脱敏方式读取；该授权不包含原始密钥、完整个人数据、配置写入或生产数据写入。
- 仓库使用只读挂载或等效的写入拒绝控制；日志和追踪系统使用无写权限的角色。配置与生产数据系统不授予凭据，也不得访问。
- “只允许只读访问”约束覆盖仓库及所有被调查系统；仅允许在当前任务的最终响应中输出报告，不得为报告创建或修改任何目标系统文件。
- 如果开始前无法证明上述只读角色、挂载或凭据边界，立即标记为 Blocked，不开始调查，也不能判定验收标准 4 通过。
- 取舍优先级依次为：安全、隐私和只读边界；固定调查范围；证据质量；指定来源覆盖；完成速度和假设数量。
- 到达 4 小时上限时必须停止，不得为了覆盖完整性突破安全、只读或时间边界；未完成内容进入完成报告。

## 事实来源

- 运行时日志优先于代码。
- 代码优先于提交信息。
- 如果来源冲突，记录冲突，不要猜测解决。

## 完成证据

1. 对验收标准 1：检查清单逐项记录两个日志来源、调用链的每一段、`v2.3` 类型；annotated tag 记录完整 tag object hash 与 peeled commit hash，lightweight tag 记录 tag ref 直接指向的完整 commit hash并注明无独立 tag object；同时记录调查开始时 `main` HEAD 完整哈希、提交选择规则和最终 20 个或不足 20 个的完整提交列表，以及对应检查和结果。无法访问的项目标记为 Unverifiable 并说明原因。
2. 对验收标准 2：发现—证据映射表逐项列出来源路径、时间戳或提交哈希，以及脱敏摘录或命令输出。
3. 对验收标准 3：报告结构检查结果，证明每项判断都有事实、假设或未知项标签，且事实与假设位于独立栏目。
4. 对验收标准 4：调查开始前记录仓库只读挂载或写入拒绝控制、incident-reader 与追踪只读角色标识及其权限清单、未配置任何配置或生产数据凭据的环境清单，并保留覆盖调查全过程的访问/命令审计。前后状态对比仅作为补充；缺少任一全过程控制证据时标记为 Unverifiable，且该验收标准不得判定通过。

## 停止与升级

- 4 小时覆盖来源读取、分析和最终报告输出的全部工作，按从首次读取任何调查来源开始的连续墙钟时间计算，包含等待授权、阻塞和暂停；报告以 UTC 记录开始时间、结束时间、总时长和停止原因。
- 最迟在 3 小时 30 分时停止新的来源读取，使用剩余最多 30 分钟仅整理此前已取得的证据并通过当前任务最终响应输出报告；总时长不得超过 4 小时。
- 交付物提前完成时立即输出报告；到达 3 小时 30 分或更早触发其他停止条件时，未完成检查和不可验证项必须进入最终报告，不得在报告定稿阶段获取新证据。
- 连续 2 次尝试没有新证据、没有减少阻塞且没有合理替代路径时停止。
- 访问 incident-reader 授权之外的敏感数据、修改任何系统或扩大时间窗口在当前 Goal 内禁止；发现需要时停止，并请求用户确认是否另建并重新审核新的 Goal，不能在当前 Goal 内继续。
