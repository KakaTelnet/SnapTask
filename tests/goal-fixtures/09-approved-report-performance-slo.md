# 审核输入内容

## Goal

使 staging 环境中 `monthly-report` 生成的 p95 延迟严格小于 30.000 秒（`p95 < 30.000s`），同时保留报表内容和 API 兼容性；恰好 30.000 秒不算通过。是否相对改动前延迟基线下降不属于完成条件。

## 背景

大型客户在报表生成超过 30 秒时会放弃等待。性能提升不能以牺牲正确性为代价。

## 验收标准

1. 在 staging benchmark 数据集上，`monthly-report` p95 在连续 3 次 benchmark 中都严格小于 30.000 秒；任一次等于或大于 30.000 秒均不通过。
2. 报表内容在总计、行数和导出 CSV 字段上匹配当前 snapshot suite。
3. 现有报表 API 契约测试保持通过。
4. 变更文件证据显示只修改了 `backend/reports/`、报表测试和 benchmark 配置。
5. 候选版本 3 次 benchmark 的计算成本中位数相对基线增长不超过 10%。

## 范围

包含：

- `backend/reports/`
- 报表测试
- benchmark 配置

不包含：

- 数据库 schema 修改
- 生产部署
- 前端报表 UI
- 计费逻辑

## 约束

- 不得修改数据库 schema。
- 不得删除报表字段。
- 以改动前提交在相同 staging 数据集、资源规格、并发和 benchmark 配置下的 3 次运行作为成本基线。
- 每次运行的计算成本以 staging 平台现有成本遥测记录的美元金额为准；增长率为 `(候选中位数 - 基线中位数) / 基线中位数 × 100%`。
- 10% 是本 Goal 内不可豁免的完成上限；超过上限时不能判定完成，用户确认也不能在当前 Goal 内放宽该上限。
- 取舍优先级依次为：安全、报表正确性和 API 兼容性；成本增长不超过 10%；p95 低于 30 秒；交付速度。

## 事实来源

- 第一优先级：当前报表 API 契约测试。
- 第二优先级：snapshot suite。
- 第三优先级：staging 平台成本遥测。
- 第四优先级：benchmark 输出。
- 如果 benchmark 收益与 snapshot 正确性冲突，保留正确性并报告冲突。

## 完成证据

- 三次 p95 严格小于 30.000 秒的 benchmark 输出，每次输出均保留用于边界判定的毫秒精度原始值。
- 映射到内容保留验收标准的 snapshot 测试输出。
- API 契约测试输出。
- 变更文件列表。
- 基线和候选各 3 次运行的成本遥测、两个中位数及增长率计算；缺少成本遥测时将验收标准 5 标记为 Unverifiable，不能判定完成。

## 停止与升级

- 所有验收标准通过且证据完成映射时完成。
- 连续 2 次尝试没有新证据、没有减少阻塞且没有合理替代路径时停止。
- 数据库 schema、生产环境、前端报表 UI、计费逻辑和数据删除在当前 Goal 内始终禁止，即使用户确认也不能在当前 Goal 内执行。
- 如果发现完成目标必须触及上述禁止项或任何其他范围外、破坏性操作，立即停止并将其标记为 Blocked 或 Out of scope；请求用户确认是否另行创建并重新审核新的 Goal，不能把确认解释为扩大当前 Goal 权限。
- 候选成本增长将超过 10% 时停止；若希望放宽上限，必须请求用户确认后创建并重新审核新的 Goal，而不是在当前 Goal 内继续。
